## Bugs Fixed

### 1. `LRUCache.mget` â€” RuntimeError: dict changed size during iteration
**File:** `storage/factory.py`  
**Severity:** High â€” crashes any batch cache read that has expired keys.  
**Root cause:** `del self._store[key]` was called inside the `for key in keys` loop while
the loop was still iterating over the same `OrderedDict`. In Python, mutating a dict during
iteration raises `RuntimeError`.  
**Fix:** Collect expired keys into a list, then delete them after the iteration completes.

---

### 2. `sync_user_md` / `sync_heartbeat_md` â€” unsafe `asyncio.create_task` from thread
**File:** `spaces/memory.py`  
**Severity:** High â€” can silently drop cache invalidations or trigger `RuntimeError` /
undefined behaviour.  
**Root cause:** `asyncio.create_task()` is **not** thread-safe. It must only be called from
the event-loop thread. These two sync functions are called from `asyncio.to_thread()`, i.e.
from a worker thread. The correct pattern is:
```python
# Wrong (can race):
loop.call_soon_threadsafe(asyncio.create_task, coro)
# Correct:
loop.call_soon_threadsafe(loop.create_task, coro)
```
**Fix:** Changed both `sync_user_md` and `sync_heartbeat_md` to use `loop.create_task`.

---

### 3. `BadgeAgent.check_and_award` â€” duplicate badge awarded in same call
**File:** `spaces/agents/gamification.py`  
**Severity:** Medium â€” causes duplicate badge names in `profile.learner.badges` when
multiple badge conditions fire in the same `check_and_award` call (e.g., first session +
beginner level both trigger on session 1).  
**Root cause:** `existing = set(profile.learner.badges)` is snapshot-frozen before the
loop. The `award()` helper appends to `profile.learner.badges` but never updates `existing`,
so later calls in the same invocation can re-award the same badge.  
**Fix:** `award()` now calls `existing.add(name)` so subsequent award checks see the badge
as already present.

---

### 4. `SpacedRepetitionAgent.update_after_review` â€” wrong SM-2 interval exponent
**File:** `spaces/agents/gamification.py`  
**Severity:** Medium â€” SRS intervals are wrong on early reviews.  
**Root cause:** `review_count` is incremented **before** computing the interval. So on the
1st recall the exponent is `2^2 = 4 days` instead of `2^1 = 2 days`. On review 0â†’1 it
would be `2^1 = 2 days` when it should be `2^0 = 1 day`.  
**Fix:** `interval_days = min(2 ** max(0, mastery.review_count - 1), 30)` â€” subtract 1 to
get a 0-based exponent after the increment.

---

### 5. `compute_next_run` duplicated in three files
**Files:** `agents/runner.py`, `agents/creator.py`, `agents/scheduler.py`  
**Severity:** Low (maintainability) â€” three identical copies; any cron-logic fix must be
applied three times.  
**Fix:**
- Added canonical `compute_next_run(schedule)` to `agents/store.py` (single location).
- `runner.py` now imports it from `store`.
- `creator.py` now imports it and aliases `_next_run = compute_next_run`.
- `scheduler.py` now imports from `store` instead of `runner`.
- Removed `datetime`/`timezone` imports from `creator.py` (now unused).
- Removed unused `import sys` from `runner.py`.

---

### 6. `WorkspaceAnalyserAgent._sample_workspace` â€” unbounded `rglob` on large repos
**File:** `spaces/agents/workspace.py`  
**Severity:** Medium â€” `ws.rglob("*")` with no depth or count limit can be extremely slow
(seconds to minutes) on repos with thousands of files (node_modules, .venv, etc.), freezing
the event loop since `_sample_workspace` is called from `asyncio.to_thread` but the loop
itself blocks if this is slow enough to saturate threads.  
**Fix:** Added depth cap (`len(rel.parts) <= 4`) and absolute count cap (200 files) so the
scan is always O(1) from the caller's perspective. The first 40 paths are still forwarded
to the LLM prompt.

### F. `agents/store.py` registry is not async-safe
The registry lock is a `threading.Lock` (correct for sync callers), but `update_agent` reads
then writes with two separate lock acquisitions, creating a TOCTOU window:
```python
spec = load_agent(agent_id)   # acquires + releases lock
...
save_agent(spec)              # acquires lock again â€” another writer could have changed spec
```
For the current usage (agent scheduler is single-event-loop, writes are rare) this is
acceptable, but worth noting for future concurrent agent runners.

### I. `cached()` helper â€” race condition on concurrent misses (thundering herd)
`storage/factory.py:cached()` has no locking, so concurrent coroutines that all miss the
cache will all call `factory()` simultaneously. For expensive LLM-backed factories (e.g.
`activity_insights`), add a per-key asyncio.Lock to serialize concurrent misses.

```python
_cache_miss_locks: dict[str, asyncio.Lock] = {}

async def cached(key, factory, ttl=None):
    cache = get_cache()
    hit = await cache.get(key)
    if hit is not None:
        ...
    lock = _cache_miss_locks.setdefault(key, asyncio.Lock())
    async with lock:
        hit = await cache.get(key)   # double-check after acquiring lock
        if hit is not None:
            return ...
        value = await factory()
        await cache.set(key, [value], ttl)
        return value
```

### J. `config.py` `_CONFIG_CACHE` â€” unbounded dict
The cache keyed by `Path` object never evicts old entries. If `load_config` is called with
many different `path` arguments (unusual but possible in tests) the cache grows without
bound. Add a max-size eviction or use `functools.lru_cache`.
